If 4-5 developers push code manually then code will conflict. One developer may overwrite another's change. No automated testing before deployment. No rollback strategies.

It means the code runs successfully on the developer's local environment but fails in any other system or environment.

1-2 times at most or even less as it is slow, risky and stressful.

Task 2:-

CI- Developers frequently merge code into a shared repository. Each push automatically triggers builds and tests.

CD- Every successful change is automatically deployed to production with no manual approval.
Deployment = Fully automated release.

CD- After CI passes, the application is automatically prepared for release.
It:
Builds artifacts
Creates Docker images
Deploys to staging
Waits for manual approval before production
Delivery = Always ready to deploy.

Task-3:- 

1) Trigger
Event that starts the pipeline.
Examples:
push
pull_request
schedule
workflow_dispatch

2) Stage
A logical group of related jobs.
Examples:
Build
Test
Deploy

3) Job
A unit of work executed on a runner.
Example:
Build Docker image
Run test suite
🔹 Step
A single command inside a job.
Example:
- run: npm install

4) Runner
The machine that executes the job.
Can be:
GitHub-hosted runner
Self-hosted runner

5) Artifact
File or output generated by a job.

Examples:
Compiled binary
Docker image
Test report
Zip package
